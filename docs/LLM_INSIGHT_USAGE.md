# LLM Insight 使用指南

## 功能说明

`/insight-llm` 命令使用 LLM 来分析 AI Agent 的调用质量，提供深度的优化建议。

## 与 /insight 的区别

| 特性 | /insight | /insight-llm |
|------|----------|--------------|
| 分析方式 | 规则算法 | LLM 分析 |
| 分析内容 | 性能指标 | 调用质量 |
| 速度 | ⚡ 毫秒级 | 🐌 秒级 |
| 成本 | 💰 免费 | 💸 消耗 token |
| 分析深度 | 📊 统计数据 | 🧠 智能洞察 |

## 使用方法

### 在 REPL 中使用

```bash
# 启动 Agent
python -m backend.main

# 执行一些任务
agent >> 帮我分析项目结构
agent >> 我想查询股票信息

# 使用 LLM 分析调用质量
agent >> /insight-llm
```

## 分析内容

### 1. 决策质量
- LLM 的工具选择是否合理？
- 是否有更好的方案？
- 是否应该调用其他工具？

**示例：**
```
决策质量：LLM 选择调用 list_dir 合理，因用户需求模糊，需先了解项目结构。
但可考虑结合历史上下文判断是否已有足够信息，避免盲目调用。
```

### 2. 效率分析
- 是否有冗余或无效的工具调用？
- 是否可以减少调用次数？
- 是否可以并行化？

**示例：**
```
效率：存在冗余——两次 web_search 和 search_lead 功能重叠，
read_file 用途不明，总耗时过长（超4分钟）。
```

### 3. 响应质量
- LLM 的回答是否准确？
- 是否完整？
- 是否有帮助？

**示例：**
```
响应质量：回答准确指出了信息缺失点，内容完整且有帮助，
引导用户补充关键细节，符合良好交互原则。
```

### 4. 提示词优化
- 用户的提示词是否可以改进？
- 如何提供更明确的信息？

**示例：**
```
提示词优化：用户提示过于简略（"帮我加个功能"），
应鼓励其说明具体功能、目标文件或使用场景，
如"在 v10_agent.py 中增加股票查询功能"。
```

### 5. 优化建议
- 具体的改进措施
- 可操作的步骤

**示例：**
```
优化建议：
- 增加上下文记忆，避免重复列出已知文件
- 若目录结构复杂，可分步交互（先确认模块，再细化功能）
- 优化工具调用性能，减少 list_dir 的响应延迟
```

## 完整示例

```bash
agent >> /insight-llm

================================================================================
🧠 LLM 调用质量分析
================================================================================

📋 运行 1/4 (ID: 683b5114)
   耗时: 8.22s | 工具调用: 0

   🤔 分析中...
   ────────────────────────────────────────────────────────────────────────────
   1. **决策质量**：LLM 未调用任何工具是合理的，因为用户请求"添加一个功能"
      过于模糊，缺乏上下文。但应主动引导用户提供更明确信息。

   2. **效率**：无冗余调用，效率尚可。但8.22秒响应偏慢，可能因模型在模糊
      输入下反复推理，建议设置快速澄清机制。

   3. **响应质量**：回答不准确且有误导性——用户并未提供"工具调用结果"，
      却要求其提供，逻辑矛盾。

   4. **提示词优化**：用户提示过于简略。建议明确说明要添加功能的系统/模块、
      功能描述、预期效果等。

   5. **优化建议**：
      - 在Agent设计中加入对模糊请求的主动澄清流程
      - 修改响应模板，避免预设用户已提供不存在的信息
      - 设置超时或快速追问机制，提升交互效率
   ────────────────────────────────────────────────────────────────────────────

================================================================================
💡 总体优化建议
================================================================================

  1. **减少冗余或低效的工具调用**
     平均每次运行调用 1.5 次工具，但其中 25% 的运行未使用任何工具。
     建议审查无工具调用的运行，确认是否因缺乏必要信息而影响结果质量。

  2. **优化 Web 搜索策略**
     web_search 是最常用工具，建议引入更精确的查询构造机制，
     设置搜索结果过滤或摘要步骤。

  3. **引入工具调用决策阈值机制**
     在 Agent 决策流程中加入置信度评估：仅当内部知识不足以高置信度
     回答时才调用工具。

  4. **监控并缩短平均响应时间（82.5 秒）**
     分析耗时瓶颈，对高频任务实施缓存或异步处理，减少等待时间。
```

## 实际应用场景

### 场景 1: 发现提示词问题

**问题：** 用户提示 "帮我加个功能"

**LLM 分析：**
```
提示词优化：用户提示过于简略。建议明确说明要添加功能的系统/模块、
功能描述、预期效果等，例如："请为我的待办事项应用添加'重复任务'功能"。
```

**改进后：** "在 backend/app/tools/ 创建 stock_tool.py，实现股票查询功能"

### 场景 2: 发现工具选择问题

**问题：** 调用了 2 次 web_search，但结果重复

**LLM 分析：**
```
效率：存在冗余——两次 web_search 和 search_lead 功能重叠，
应合并搜索请求，避免重复。
```

**改进：** 优化搜索策略，一次性获取所需信息

### 场景 3: 发现响应质量问题

**问题：** LLM 回答不准确

**LLM 分析：**
```
响应质量：回答不准确且有误导性——用户并未提供"工具调用结果"，
却要求其提供，逻辑矛盾。应改为直接询问功能细节。
```

**改进：** 修改 Agent 的系统提示词，改进响应逻辑

## 成本说明

`/insight-llm` 会调用 LLM API 进行分析，会消耗 token：

- **每次运行分析**: 约 500-1000 tokens
- **总体建议**: 约 300-500 tokens
- **总成本**: 取决于运行次数和 LLM 模型

**示例：**
- 4 次运行 = 约 4000-5000 tokens
- 使用 DeepSeek: 约 ¥0.01-0.02
- 使用 Claude: 约 $0.02-0.05

## 与其他工具的配合

### 工作流程

```
1. 执行任务
   agent >> 帮我分析项目结构

2. 性能分析（快速）
   agent >> /insight
   → 发现耗时 24.6s，有 1 次工具调用

3. 质量分析（深度）
   agent >> /insight-llm
   → 发现工具选择合理，但提示词可以优化

4. 应用优化
   agent >> 请分析 backend/ 目录的 Python 文件结构

5. 验证效果
   agent >> /insight
   → 耗时降低到 5.2s
```

## 注意事项

1. **成本**: 会消耗 API token，建议在需要时使用
2. **速度**: 比 `/insight` 慢，需要等待 LLM 分析
3. **准确性**: LLM 分析可能不完全准确，需要人工判断
4. **隐私**: 会将 trace 数据发送给 LLM API

## 故障排查

### 问题: 分析失败

**原因**: LLM API 调用失败

**解决**: 检查 API 密钥和网络连接

### 问题: 分析结果不准确

**原因**: LLM 理解有偏差

**解决**: 结合 `/insight` 的数据指标综合判断
